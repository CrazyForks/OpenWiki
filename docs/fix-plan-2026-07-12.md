## 批次 B：核心 Bug 修复（7 个，预计 2-3 天）

### B1. Wiki 自动编译失败后永远不会重试【严重·数据完整性】
**位置**：`src-tauri/src/ai/wiki_engine.rs:623`（失败分支），入口守卫在同文件 `:570`。
**现状**：编译失败分支写着注释 "Don't update compile_hash on failure — will retry next time"，但同一行却调用了 `update_content_assessed_hash(content_id, &current_hash)`。而入口处只要 `wiki_assessed_hash == current_hash` 就直接 `return Ok(())`。结果：AI 接口超时/断网失败一次，这条内容**永远**不会再被自动编译进知识库，且无任何提示。注释与行为直接矛盾。
**改法**：
1. 删除失败分支中的 `update_content_assessed_hash` 调用，使行为与注释一致。
2. **必须先调研重试频率**：找到自动编译的触发点（读 `src-tauri/src/scheduler/` 和 `lib.rs` 中的定时任务），确认失败后不会以秒级频率疯狂重试烧 API。若触发频率高于每小时一次，需加简单退避：在内存（如 `HashMap<content_id, fail_count>`）记录失败次数，同一条内容连续失败 3 次后本次进程内不再重试（下次启动自然重试）。
3. 为该逻辑补一个单元测试：模拟编译失败 → 断言 `wiki_assessed_hash` 未被更新。
**验收标准**：失败分支不再更新 hash；有对应单测；执行者在 commit 消息中说明实际重试频率与退避策略。

### B2. 每次启动清空 "related" 边表，Q&A 页面的图谱连线被永久删除【严重·数据丢失】
**位置**：`src-tauri/src/lib.rs:159`（`delete_edges_by_relation("related")`，注释自称"一次性迁移"但每次启动都执行）；Q&A 边的创建在 `src-tauri/src/commands/wiki.rs:925-926`（`relation="related"`，且该页面 `tags: None` 见 `:900`）；标签边的自动重建在 `src-tauri/src/ai/wiki_engine.rs:769-771`（按 tags 过滤，因此 Q&A 边不会被重建）。
**现状**：普通标签边启动后 5 秒会被 TF-IDF 重建，但用户在"问一问"里保存的问答页引用连线删了就再也回不来。
**改法**：
1. 把这次清理变成真正的一次性迁移：用 preferences 表存一个标志（如 `migration_related_edges_cleared = "true"`），启动时检查，已执行过则跳过。参考 repository 中现有的 get_setting/set_setting 用法。
2. **加分项（先调研再做）**：检查已保存的 Q&A 页面数据中是否保留了来源 content id（读 `wiki.rs` 保存 Q&A 页面的完整逻辑和页面数据结构）。若有，写一段一次性补偿逻辑：启动时扫描 Q&A 页面，为缺失的引用边重建 `related` 边。若数据不足以重建，在 commit 消息中说明"历史丢失无法恢复"。
**验收标准**：连续启动两次（或单测模拟），第二次不再执行 `delete_edges_by_relation`；保存一个 Q&A 页面 → 重启 → 该页面的图谱连线仍在。

### B3. 剪贴板图片孤儿文件：磁盘无限增长 + 隐私风险【严重·隐私】
**位置**：`src-tauri/src/capture/clipboard.rs:114`（剪贴板一出现新图就先落盘 `captures/{uuid}.png`，发生在去重/用户确认**之前**）；`src-tauri/src/commands/capture.rs:521` `fn copy_image_to_captures`（用户确认保存时**又复制一份** `captures/{id}.png` 入库，原始文件从不删除）；仅 `dismiss_capture` 在用户主动取消时清理一次。
**现状**：用户拒绝保存、超时消失、被去重抑制的图片（可能含敏感截图）全部以明文 PNG 永久留在数据目录。敏感词过滤只查文本（`capture/detector.rs:221`），对图片无效。
**改法**（分两步，都做）：
1. **止血——不再产生新孤儿**：梳理临时图片文件的完整生命周期（落盘 → 浮窗预览 → 确认/取消/超时/去重抑制）。目标：**每一个出口都处理掉临时文件**——确认保存时优先用 `fs::rename`（move）代替 copy（注意跨卷 rename 会失败，需 fallback 为 copy+delete）；取消、超时、去重抑制路径全部删除临时文件。注意浮窗可能仍持有该文件路径用于预览，删除时机要在浮窗关闭之后。
2. **历史孤儿安全清理**：新增一个启动后延迟执行的清理任务：扫描 captures 目录，找出**不被数据库任何记录引用**的图片文件。⚠️ 安全要求：先读 repository 确认数据库中图片路径的存储格式（绝对路径/相对路径/仅文件名），匹配逻辑必须保守（宁可漏清不可误清）；找到的孤儿**不物理删除**，统一移入 `captures/_orphaned/` 子目录并在日志记录数量；`_orphaned/` 目录本身要被清理扫描排除。
**验收标准**：复制图片→浮窗上不点保存（等它消失）→ captures 目录无新增常驻文件；复制图片→保存→ captures 目录只有 1 个对应文件；启动后历史孤儿被移入 `_orphaned/`，数据库在用的图片一个不少（界面上历史图片仍能正常显示）。

### B4. 内容列表筛选/计数只作用于已加载的分页【严重·大库不可用】
**位置**：`src/features/content-list/ContentList.tsx`（v0.3.21 行号）：客户端 filter 逻辑在组件内（v0.3.20 时为 464-487 一带，grep `setFilter` / filter 相关 useMemo 定位）；tab 计数与 `count === 0` 隐藏逻辑约 `:995`；滚动触底加载 `loadMore`。后端查询在 `src-tauri/src/storage/repository.rs`（`get_all_content(limit, offset)`）与 `src-tauri/src/commands/storage.rs`。
**现状**：列表按 50 条/页无限滚动，但"文本/图片/链接/分类/时间"筛选只在**已加载的前 N 条**里过滤。几千条记录的用户点"图片"只能看到已加载的那几张；筛选结果太短没有滚动条时 `loadMore` 永不触发；各 tab 计数来自已加载数据、"全部"却用 DB 总数，数字互相矛盾，筛选 tab 还会随滚动"凭空出现"。
**改法**（本任务是本批最大的一个，允许中型重构，但范围限定在列表数据流）：
1. 后端：为内容查询命令增加筛选参数（类型 type、分类 category、时间范围 range），SQL 层 WHERE + LIMIT/OFFSET 分页；同时提供一个按同样筛选条件的 `count` 查询（各类型/分类的计数用 GROUP BY 一次取回）。全部走参数化 SQL（项目现有代码全是参数化的，保持一致）。
2. 前端：筛选变化 → 重置分页并从服务端拉第一页；滚动触底 → 带着当前筛选拉下一页；tab 计数改用服务端 count 结果；删除客户端 filter 逻辑。
3. **本任务不做虚拟化**（列表 DOM 优化另行考虑），只修正确性。
4. 兼容既有功能：批量导入（v0.3.21 新增，同文件内 +264 行）、高亮、多选删除等现有行为不得回归。
**验收标准**：用脚本向 SQLite 插入 300 条混合类型测试数据（或执行者自写临时脚本）后：点"图片"筛选能滚动加载到**全部**图片条目；各 tab 计数从首屏起就与数据库实际数量一致且不随滚动变化；筛选后内容不足一屏时也能正确显示全部结果。

### B5. 窗口失焦再回焦，列表被截断回 50 条【严重·高频触发】
**位置**：`src/features/content-list/ContentList.tsx:562-566`（v0.3.21）：`window` focus 事件无条件 `loadInitial()`，`setContents(data)` 用第一页整体替换已加载的全部数据。
**现状**：用户滚到第 500 条，cmd-Tab 切走再回来，列表瞬间只剩 50 条、滚动位置跳变。桌面应用切窗口极频繁，几乎每天发生。
**改法**（在 B4 完成后做）：focus 回来时的刷新改为"保持当前加载量"：重新请求 `offset 0 .. 当前已加载条数`（一次查询，limit=当前长度），替换数据但长度不缩水；或对比第一页数据无变化则不动。保持滚动位置不跳。
**验收标准**：加载 3 页（150 条）→ 切走再切回 → 列表仍是 150 条且滚动位置不变；期间新捕获的内容在回焦后出现在顶部。

### B6. 搜索结果点击后对未加载的旧条目完全失效【严重】
**位置**：`src/App.tsx:293` 一带（点搜索结果 → `switchTab("content", [item.id])`）；`src/features/content-list/ContentList.tsx:624-641`（v0.3.21，scrollToId effect 只在已渲染的 `cardRefs` 里找 DOM）。
**现状**：搜索是全库的，但列表只加载了前几页。点一条上月的记录 → 切到内容页后不滚动、不高亮、不加载，毫无反应。附带 bug：目标不存在时 `scrollToId` 永不清空，后续任何列表更新都会重复执行 `setFilter("all")` 重置用户的筛选。
**改法**（依赖 B4 的服务端分页）：
1. scrollToId 指向的条目未在已加载列表中时：调用后端按 id 定位（方案任选：新增"加载到目标 id 所在页"的查询，或单独取该条并插入顶部+高亮）。推荐前者（保持列表连续性）。
2. 无论成功失败，处理完毕后清空 `scrollToId`，不再残留副作用。
**验收标准**：插入 300 条数据 → 搜索一条第 250 位的内容 → 点击 → 切到内容页后该条可见并高亮；此后手动切换筛选不再被自动重置。

### B7. 知识图谱 rAF 绘制循环永不停歇，后台持续烧 CPU【严重·性能】
**位置**：`src/features/wiki/WikiGraphView.tsx:176-179`（`tick` 无条件 `requestAnimationFrame` 自续，d3 力导模拟冷却后也不停）；`src/App.tsx:367-383` 一带（所有主 tab 常驻挂载，仅 `display:none` 隐藏）。
**现状**：打开过"知识库→图谱"后切去别的 tab，画布仍以 60fps 全量重绘，CPU/电量持续消耗。
**改法**（两个方向选其一或组合）：
1. 模拟冷却后停画：监听 d3 simulation 的 alpha（`alpha() < alphaMin()`）或 `end` 事件 → 停止 rAF 循环；用户交互（drag/zoom/pan/hover）或数据变化时重启循环。
2. 不可见即暂停：组件容器挂 IntersectionObserver（`display:none` 时不相交），不可见 → 暂停 rAF，重新可见 → 恢复。
推荐两个都做（1 解决"图谱页开着不动也烧 CPU"，2 解决"切走了还在画"）。
**验收标准**：在 tick 回调临时加 `console.count`（验收时移除或用性能面板）：图谱布局稳定后计数停止增长；切到其他 tab 后计数停止增长；拖拽节点时恢复流畅。
